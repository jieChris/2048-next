# 2048-next 项目完善任务文档（完整版）

更新时间：2026-03-15  
适用分支：`main`（基于当前仓库快照）  
文档目标：为后续 2-4 个迭代提供“可执行、可验收、可裁剪”的系统改造任务清单。

---

## 1. 执行目标与边界

### 1.1 目标
- 在不破坏现有玩法与线上稳定性的前提下，降低维护成本与复杂度。
- 收敛双栈（TS + legacy JS runtime）长期并存导致的架构负担。
- 建立稳定核心边界（状态/规则/回放/存档契约）与可持续测试治理机制。
- 降低页面 CPU/内存开销与首屏资源负担。

### 1.2 非目标
- 不做大规模视觉重构（UI 风格不作为当前阶段主目标）。
- 不引入后端重写（仅对 API 使用与容错策略做前端治理）。
- 不在单次迭代内一次性移除全部 legacy runtime（采用分阶段替换）。

## 2. 当前状态快照（基于仓库实际扫描）

### 2.1 规模与结构
- HTML 页面入口：12 个（`vite.config.ts` 多入口构建）。
- `src`（TS）文件：139，约 19006 行。
- `js`（legacy/runtime）文件：172，约 35828 行。
- 测试文件总数：154（unit 125 + smoke 28 + 其他）。

### 2.2 复杂度高位文件（行数）
- `js/core_game_manager_replay_helpers_runtime.js`：2394 行
- `js/core_game_manager_move_input_helpers_runtime.js`：1185 行
- `js/core_game_manager_saved_state_helpers_runtime.js`：1132 行
- `js/core_game_manager_mode_rules_helpers_runtime.js`：1071 行
- `js/online_leaderboard_runtime.js`：797 行

### 2.3 架构现状关键事实
- `src/core/engine.ts` 当前仍是轻量壳（尚未成为真实统一引擎入口）。
- 业务入口大量依赖 `src/entries/*` 中的 `loadLegacyScriptsSequentially(...)` 串行加载 legacy 脚本。
- 历史记录仍主要使用 `localStorage` 全量读写模式（`js/local_history_store.js`）。
- 已有较强测试体系，但重构门禁与复杂度门槛规则刚性较高，维护摩擦上升。

## 3. 问题总览（结论）

### 3.1 架构层
1. 双栈长期共存，认知和排障成本高。  
2. 核心边界未完全收口，核心对象（Engine）职责偏弱。  
3. 多入口 + 多 bootstrap 路径造成重复与一致性风险。  
4. 全局对象耦合较重（`window.*` + runtime 注入链）。

### 3.2 性能与资源层
1. 历史数据采用 `localStorage` 全量序列化，记录增长时主线程压力明显。  
2. 若干大体积静态资源（字体/图标）对首屏加载和内存有压力。  
3. 多页面存在各自轮询/刷新策略，统一调度体系仍不完整。

### 3.3 测试与工程治理层
1. 测试规模大，后续执行时间与 flaky 管理成本会上升。  
2. 门禁规则（如函数行数热区）对重构质量有帮助，但需防止“规则驱动代码变形”。  
3. 文档体系里“推进记录”重、“长期架构文档”轻，可读性和交接效率不足。

### 3.4 数据契约层
1. replay / undo / history / online 提交协议分布在多层实现，缺少单一契约源。  
2. 版本迁移策略不够集中，易出现页面层和核心层各自演进。

## 4. 详细问题与改造方向（按域拆解）

## 4.1 架构与模块边界

### 问题 A1：`Engine` 抽象尚未形成真正核心边界
- 现状位置：`src/core/engine.ts`
- 风险：
  - 规则执行、状态迁移、回放执行路径仍分散。
  - 难以建立“引擎层可替换、页面层可简化”的结构。
- 目标：
  - 建立稳定 `Engine API`：`init/loadState/move/undo/replayStep/exportState/importReplay`。
- 任务：
  - 输出引擎状态模型与命令模型（严格类型定义）。
  - 将 move/undo/replay 的核心逻辑统一从 runtime helpers 回收至 `src/core`。
  - 页面只做 IO 与渲染，不再持有规则判断。

### 问题 A2：入口脚本链过长且重复
- 现状位置：`src/entries/play.ts`（大量 URL 导入 + 脚本数组）及其他 entries。
- 风险：
  - 每新增 runtime 都要多点维护，漏改概率高。
  - 启动链可读性低，调试成本高。
- 目标：
  - 由“手写脚本数组”改为“声明式 manifest + 校验器”。
- 任务：
  - 建立 `runtime-manifest.ts`，按页面定义依赖组。
  - 自动校验 `manifest` 与实际文件存在性、顺序约束。
  - 后续逐步将 manifest 条目替换为原生 TS 模块依赖，最终移除串行 loader。

### 问题 A3：多入口页面长期维护负担高
- 现状位置：根目录 12 个 HTML + `vite.config.ts` 多入口。
- 风险：
  - 元信息、注入配置、初始化流程容易漂移。
  - 新功能常需多页重复接入。
- 目标：
  - “入口可保留，启动流程模板化”。
- 任务：
  - 抽象统一 page bootstrap 模板（按模式/功能差异注入参数）。
  - 统一页面级 capability 配置（功能开关、是否需要在线榜单、是否支持回放）。
  - 对仅内部调试页面建立 `dev-only` 入口策略，避免生产包冗余。

---

## 4.2 性能、内存与资源治理

### 问题 B1：历史存储路径低效（全量 parse/stringify）
- 现状位置：`js/local_history_store.js`
- 风险：
  - 记录多时会出现明显主线程卡顿与内存峰值上升。
  - 分页查询仍依赖内存中全量数组筛选排序。
- 目标：
  - 从 `localStorage` 迁移至 `IndexedDB`（分片、索引、增量查询）。
- 任务：
  - 设计历史记录表结构（主键、mode、time、score 索引）。
  - 提供兼容迁移器：首次启动把旧数据搬迁到 IndexedDB 并打迁移标记。
  - `list/getById/export/import` 改成异步 DAO，支持游标分页。

### 问题 B2：静态资源体积偏大
- 现状位置：`dist/assets`（当前存在 MB 级字体与大体积 SVG）
- 风险：
  - 首屏下载和解析开销高。
  - 低端设备内存压力增加。
- 目标：
  - 建立资源预算（字体、logo、icon 的上限）与替代策略。
- 任务：
  - 字体：子集化/按需加载，减少全量 TTF 常驻。
  - 图标：大 SVG 进行压缩或替换为更轻量版本。
  - 加入构建体积告警（超预算阻断或告警）。

### 问题 B3：刷新调度分散
- 现状位置：各页面 runtime（排行榜、回放、定时 UI）
- 风险：
  - 页面后台仍可能保留不必要刷新。
  - 不同模块各自调度，难统一优化策略。
- 目标：
  - 建立“可见性 + 用户活跃状态 + 错误退避”的统一刷新策略。
- 任务：
  - 封装公共 scheduler 工具（支持前后台分频、退避、立即唤醒）。
  - 对回放、榜单、计时 UI 统一迁移。

## 4.3 数据契约与协议统一

### 问题 C1：replay/history/undo 契约未集中管理
- 现状位置：
  - `src/core/replay-codec.ts`
  - `js/local_history_store.js`
  - `js/core_game_manager_*_runtime.js` 多处相关逻辑
- 风险：
  - 改字段时容易漏改，出现兼容性断裂。
  - 数据格式演进缺少“单点版本策略”。
- 目标：
  - 建立统一 `contracts/`：ReplayRecord、HistoryRecord、SessionSnapshot、SubmitPayload。
- 任务：
  - 统一 schema 版本号与升级器（v1 -> v2 ...）。
  - 所有入口通过统一 adapter 读写，不直接拼装对象。
  - 引入契约测试（schema round-trip + backward compatibility）。

### 问题 C2：在线榜单提交与渲染契约分散
- 现状位置：`js/online_leaderboard_runtime.js`、`js/account_page.js`
- 风险：
  - 文案、字段、错误处理逻辑重复。
  - 线上字段调整会引发多处连锁改动。
- 目标：
  - 抽出单一 leaderboard client + view model mapper。
- 任务：
  - API 请求、重试、timeout、fallback 统一在 client 层。
  - 页面仅消费标准化 `LeaderboardRowViewModel`。

## 4.4 测试与质量门禁治理

### 问题 D1：测试规模增长后执行成本上升
- 现状：unit + smoke 已较大。
- 风险：
  - 提交等待时间延长，影响迭代节奏。
  - 烟测失败定位成本逐步增加。
- 目标：
  - 分层执行策略：PR 快速集 + 合并前完整集 + 夜间全量回归。
- 任务：
  - 定义 `tier-1/tier-2/tier-3` 测试清单及触发条件。
  - 引入失败聚类报告（按页面/能力分类）。
  - 建立 flaky 白名单和自动重试策略（仅针对已确认 flaky 用例）。

### 问题 D2：复杂度门禁过于“行数导向”
- 现状位置：`scripts/game-manager-audit.mjs`
- 风险：
  - 可能诱发“为过门禁而切碎函数”，可读性反而下降。
- 目标：
  - 由单一行数约束扩展为复合指标（复杂度、重复率、依赖扇出）。
- 任务：
  - 保留行数阈值，但新增：圈复杂度上限、重复片段检测、模块耦合评分。
  - 审计结果输出“修复建议”而非仅失败。

## 5. 需要替代或移除的低效/无效代码清单（重点）

> 说明：以下为“目标处置清单”，并非要求一次性删除。建议按阶段迁移后再移除。

1. `src/entries/*` 中手工维护的超长脚本导入与数组拼装  
- 类型：低效维护代码  
- 处置：替换为 manifest 驱动 + 自动校验；迁移完成后减少重复入口脚本声明。

2. `src/entries/legacy-loader.ts` 的长期依赖  
- 类型：过渡层代码（阶段性合理，长期应退出）  
- 处置：核心能力 TS 化完成后，逐页移除串行 legacy 加载。

3. `js/local_history_store.js` 全量 `localStorage` 读写路径  
- 类型：高成本实现  
- 处置：替换为 IndexedDB DAO；保留一次性迁移器；迁移稳定后移除旧实现。

4. `js/core_game_manager_replay_helpers_runtime.js` 超大聚合模块  
- 类型：高复杂度热点  
- 处置：按职责拆分（codec/import/execution/session bridge），并将纯算法迁入 `src/core`。

5. `scripts/game-manager-audit.mjs` 中单一“函数行数”强门禁  
- 类型：治理策略偏单一（不是无效代码）  
- 处置：升级为复合质量指标，避免规则反向塑形。

6. `REFACTOR_MANAGEMENT_PLAN.md` 的超长推进记录主文档形式  
- 类型：低可维护知识载体  
- 处置：拆分为架构文档 + 迁移清单 + 历史归档，主文档轻量化。

7. 仅用于开发调试且不应进入生产构建路径的页面/入口（如测试入口）  
- 类型：潜在冗余产物  
- 处置：建立 `dev-only` 标识与构建排除策略，减少生产构建体积和维护负担。

## 8. 验收标准（全局）

### 8.1 架构验收
- 核心引擎 API 文档齐全，页面层不再直接依赖深层规则实现。
- legacy loader 使用范围明确缩减，并有退场计划与里程碑。

### 8.2 性能验收
- 历史页大数据集场景无明显卡顿（分页/筛选响应稳定）。
- 后台标签页刷新频率明显下降，网络请求有节制。
- 关键静态资源满足体积预算。

### 8.3 质量验收
- unit + smoke 全绿。
- 门禁规则对问题定位有可解释输出。
- 关键数据契约变更具备迁移测试与回放兼容测试。

### 8.4 文档验收
- 架构文档、迁移清单、运维手册结构清晰、可交接。
- 超长推进记录完成归档，主文档保持简洁可读。

## 9. 风险与回滚策略

### 9.1 主要风险
- 迁移过程中出现 replay/history 不兼容。
- 双栈切换阶段引入隐性行为差异。
- 测试执行成本短期上升，影响开发节奏。

### 9.2 回滚策略
- 对关键能力引入 feature flag（引擎路径、存储路径、刷新策略）。
- 每阶段保留可回滚版本标签。
- 先并行灰度，再默认切换，最终移除旧路径。

## 12. 给 Agent 的执行提示词（可直接复制）

### 12.1 一次性交付模式（推荐）

```text
你是该仓库的执行 Agent。请基于根目录 PROJECT_IMPROVEMENT_TASKS.md 一次性推进到“可交付标准”，而不是只做基线分析。

执行要求：
1) 先输出执行计划（按 P0->P5 拆解，标注每步影响文件、风险、验收条件），然后立即开始实施，不要停在方案阶段。
2) 默认自主推进：除非遇到不可解阻塞，不要反复向我提问。
3) 每完成一个子任务必须做“自检+测试+修复回路”：
   - 自检：代码风格、类型、重复逻辑、兼容性、性能回退风险。
   - 测试最小集：npm run build && npm run test:unit
   - 涉及页面交互/回放/历史/模式入口时：追加 npm run test:smoke:ci
   - 如某门禁脚本失败但与本次改动无关，需说明原因并继续执行可验证测试，不得直接中断。
4) 自动修复机制：如果测试失败，自动定位并修复，重复“修复->重测”最多 3 轮；3 轮后仍失败再汇报阻塞点。
5) 所有改动完成后，输出最终交付报告，包含：
   - 变更摘要（按模块）
   - 关键文件列表
   - 测试命令与结果（通过/失败 + 原因）
   - 已知风险与后续建议
6) 若没有明确要求不要提交：在测试通过后自动提交，提交信息需反映本轮主题。

交付标准：
- 满足文档第 8 章“验收标准（全局）”；
- 关键路径测试全绿或给出可复现、可定位的阻塞说明；
- 输出可直接用于发布/验收的报告。
```

### 12.2 分阶段自动推进模式（更稳妥）

```text
请按 PROJECT_IMPROVEMENT_TASKS.md 分阶段执行：
阶段顺序：P0 -> P1 -> P2 -> P3 -> P4 -> P5。

每阶段都必须执行：
1) 实施改动
2) 自动自检
3) 自动测试（build + unit；必要时 smoke）
4) 自动修复失败项（最多 3 轮）
5) 产出阶段报告（完成项/未完成项/风险/下阶段计划）

规则：
- 阶段未达到验收条件不得进入下一阶段；
- 除非阻塞，不要等待人工确认；
- 遇到阻塞先给出“可选解决路径 + 推荐路径”，并默认执行推荐路径。
```

### 12.3 问题发现与自动检查提示词（补充）

```text
在整个执行过程中，持续进行问题扫描，不限于当前需求文件：
- 检查重复代码、死代码、过时兼容分支、无效轮询、低效存储路径、超大静态资源。
- 对每个发现的问题给出：影响范围、风险等级、修复建议、是否本轮处理。
- 本轮不处理的问题，必须登记到“待办风险清单”并给出触发条件。
```
